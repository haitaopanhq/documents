# AI-Native Resume / AI 原生简历包

这套简历的唯一主线是：**用 AI 原生方式，把业务知识、工具调用、工作流和可靠交付连接成可使用的产品。**

当前的产品组合是这条主线最直接的证据：**Xstream App**（AI-native 客户端产品实践）、**Xworkmate / AIworkspace**（AI 协作与知识工作空间）和 **[svc.plus](https://www.svc.plus)**（Cloud-Neutral Toolkit、文档与产品入口）。三者合起来呈现“应用 → 工作空间 → 开放平台”的产品闭环，而不是传统基础设施工具集。

它不投传统运维/SRE、纯 AI Infra、外包、驻场或外派岗位。Kubernetes、IaC、可观测性等只作为把 AI 应用安全地接入企业环境的工程能力出现，不作为职业标签。

## 目标岗位

| 优先级 | 中文岗位 | English title | 核心匹配 |
|---|---|---|---|
| A | AI 原生应用工程师 / Agent 工程师 | Applied AI Engineer / Agent Engineer | Agent 工作流、MCP、RAG、工具集成、可评估交付 |
| A | AI 解决方案架构师（应用方向） | AI Solutions Architect, Applied AI | 业务发现、PoC、私有化/企业集成、客户价值 |
| B | AI 产品工程师 / 创始工程师 | AI Product Engineer / Founding AI Engineer | 从问题到可用产品、全栈交付、AI coding |
| B | AI 实施顾问 / 客户工程师 | AI Implementation Consultant / Customer Engineer | 企业知识、流程自动化、落地与培训 |

## 不投与不写

- 职位：SRE、DevOps、运维、NOC、纯 AI Infra、GPU 集群运维、MSP 驻场、外包/派遣。
- 标题：不要出现 “AI Ops Engineer”“AI Infrastructure Specialist”“运维平台工程师”。
- 叙事：不以告警、值班、故障处理或集群维护开场；除非 JD 明确需要，否则不要把工具清单放在摘要前。
- 未经核验的表述：将“生产上线”“准确率 85%+”“成本降低 60%”“3 个月上线”等统一放入证据核验表；未能提供项目材料、客户确认或可复现测量时，改写为“PoC / 方案设计 / 验证”。

## 使用方式

1. 每次投递只选一个主标题；A 类岗位用 `AI 原生应用工程师 / Applied AI Engineer`，偏客户场景时用 `AI 解决方案架构师 / AI Solutions Architect`。
2. 以 `01` / `02` 为母稿，按 JD 替换“精选项目”的顺序和关键词；保持一页，最多两个项目。
3. 把可点击的公开代码、Demo、文档和架构图放入 `03`，不要在主简历堆砌工具名。
4. 每次投递前按 `05` 核验所有数字、客户名、职责和项目阶段。
